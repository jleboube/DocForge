# Authentication and Tenant Isolation

## Implemented
- Google OAuth authorization code flow in web UI.
- OAuth endpoints:
  - `GET /auth/google/start`
  - `GET /auth/google/callback`
  - `POST /auth/google` (legacy/manual token exchange path)
- App JWT session token issued by API and sent as `Authorization: Bearer <token>`.
- Tenant isolation enforced with `userId` filters across:
  - `sources`
  - `documents`
  - `document_contents`
  - `document_chunks`
  - `highlights`
  - `ingestion_jobs`
  - `source_sync_events`

## Isolation Guarantee
Authenticated users only query and mutate rows where `userId` equals their subject ID.
No endpoint exposes global user counts or cross-tenant identifiers.

## Worker Auth
Background workers call admin endpoints with `x-internal-token` header.
This prevents anonymous invocation while allowing system-level job processing.

## Secret Generation
- `JWT_SECRET` and `INTERNAL_API_TOKEN` are generated by `scripts/bootstrap-env.sh`.
- `scripts/deploy.sh` runs bootstrap then starts compose.
- External OAuth secrets (`GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`) are user-provided.

## Apple OAuth
`POST /auth/apple` is scaffolded and returns setup requirements.
Full Apple web OAuth requires Services ID, Team ID, Key ID, private key, and callback domain configuration.
